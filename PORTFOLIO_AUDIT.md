# Public Portfolio Curation Audit

This audit prepares the project for a public GitHub portfolio repository. The goal is recruiter readability, not publishing every local working artifact.

## Recommended Public Repository Structure

```text
.
├── README.md
├── LICENSE
├── .gitignore
├── PORTFOLIO_AUDIT.md
├── backend/
│   ├── .env.example
│   ├── package.json
│   ├── package.json
│   ├── .env.example
│   └── README.md
├── frontend/
│   ├── package.json
│   ├── package-lock.json
│   ├── angular.json
│   ├── tsconfig*.json
│   ├── tailwind.config.js
│   ├── postcss.config.js
│   └── src/
└── docs/
    ├── README.md
    ├── api_spec.md
    ├── current-system-audit.md
    ├── db_table_map.md
    ├── premium-report-ai-backbone.md
    ├── premium-report-persistence-implementation-v1.md
    ├── premium-report-mvp-status-v1.md
    ├── tone-system/
    └── selected architecture/product docs
```

## Include

- `backend/README.md`: redaction note and private backend architecture summary.
- `backend/.env.example`: public-safe environment variable template.
- `backend/package.json`: redacted backend package metadata retained for stack context.
- `frontend/src/app/`: Angular routes, reusable report components, auth flow, admin screens, premium report renderer, and services.
- `frontend/src/assets/placeholders/`: safe placeholder SVG assets.
- `frontend/src/assets/premium/`: include only if these images are owned/generated by the project and licensed for public display.
- `frontend/package.json`, `frontend/package-lock.json`, Angular/Tailwind/PostCSS/TypeScript config files.
- `docs/`: keep public architecture, API shape, frontend route map, tone/safety, and roadmap docs.
- `README.md`, `.gitignore`, `LICENSE`, `PORTFOLIO_AUDIT.md`.

## Exclude

- `backend/.env`: contains secrets and a live-looking OpenAI API key.
- Any `.env.*` except `.env.example`: environment-specific configuration can leak credentials and production topology.
- `backend/dist/`, `frontend/dist/`: generated build artifacts; recruiters should review source code.
- `frontend/.angular/`: Angular build cache; reinstall/rebuild locally.
- `node_modules/`: vendor dependencies that can be installed from lockfiles.
- `backend/tmp/`: generated audit outputs, raw LLM/debug artifacts, and large local analysis files; may contain user-derived or operational data.
- `backend/src/`, `backend/scripts/`, and backend logic docs: proprietary report-generation, scoring, prompt, and pattern logic.
- Uploaded user files and external upload roots such as `htp-upload/`, `uploads/`, or `storage/`: private user data risk.
- `.DS_Store` and editor/OS files: local noise.
- `tune-analyze/` and `tune_analyze/`: third-party product style-guide reference material and scraped/derived notes; replace with a short sanitized internal writing-principles document if needed.
- `legacy/`: old PHP implementation appears to be legacy/proprietary reference code with raw SQL and operational assumptions. Exclude from the public repo unless ownership is clear and you intentionally want to show migration history. If retained privately, summarize it in docs instead.

## Security Concerns Found

- `backend/.env` includes `OPENAI_API_KEY`, `DB_PASSWORD`, and `JWT_SECRET`. Do not publish it. Rotate/revoke the OpenAI key before any public push.
- Backend config currently defaults upload roots to an absolute local path under `/Users/seongrok/...`; this is not a secret, but public code should prefer environment-driven paths and documentation.
- The frontend stores JWTs in `localStorage`. This is acceptable for a portfolio MVP, but note the XSS trade-off in production.
- Admin routes expose powerful report generation and inspection flows. Keep them behind admin auth and avoid publishing production credentials or live deployment URLs.
- `backend/tmp` includes large generated JSON/HTML audit outputs that may reflect real database rows or internal evaluation data.
- The mock report asset may be acceptable only if it is synthetic/anonymized. If it came from a real user, replace it with a fully synthetic fixture.

## Third-Party Or Company/IP Review

- `tune-analyze/` includes third-party consumer health and personality product reference notes. Do not publish the raw notes unless you are certain they are original and legally safe. Prefer a sanitized `docs/report-writing-principles.md` that describes general lessons without copying protected content.
- `legacy/` may be company/proprietary code or an inherited production implementation. It is not required for recruiter comprehension because the TypeScript backend already demonstrates the rebuilt architecture. Exclude and document the migration at a high level.
- Premium visual assets should be included only if generated/owned by this project. If ownership is uncertain, replace with simple placeholders or a small set of clearly licensed demo images.

## Additional Recruiter Presentation Improvements

- Add 3-5 screenshots or a short GIF under `docs/screenshots/` using synthetic/demo data.
- Add a concise architecture diagram to the root README.
- Add a `docs/database-schema.md` or SQL migration folder with sanitized schema definitions.
- Add a `docker-compose.example.yml` for local MySQL if you want easier reviewer setup.
- Add conventional test scripts to `package.json` for backend smoke tests and frontend checks.
- Replace any real sample report fixtures with synthetic cases named `sample-premium-report.json`.
- Add a short "What I Built" section to the README that highlights backend architecture, AI pipeline design, and frontend rendering decisions.
